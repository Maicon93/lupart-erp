# i18n.md

## Estrutura de chaves

- Padrão: `modulo.contexto.CHAVE`
- Pastas/contextos em **minúsculo** (`products.errors.`, `common.actions.`)
- Chaves finais em **UPPER_SNAKE_CASE** (`NOT_ENOUGH_STOCK`, `SALE_COMPLETED`)
- Exemplos:
  - `products.fields.NAME`
  - `products.errors.NOT_ENOUGH_STOCK`
  - `sales.messages.SALE_COMPLETED`
  - `common.actions.SAVE`
  - `common.validations.REQUIRED_FIELD`

## Sincronização API ↔ Frontend

- As chaves devem ser **exatamente iguais** na API e no frontend
- A API envia apenas a chave via `messageCode` (ex: `products.errors.not_enough_stock`)
- O frontend traduz a chave recebida via i18n

## API

- Arquivo centralizado de chaves com **comentário em português** descrevendo o significado
- Exemplo:
  ```typescript
  // products.errors.NOT_ENOUGH_STOCK — Estoque insuficiente para a operação
  ```

## Frontend

- Um arquivo por idioma (`pt-BR.json`, `en.json`)
- Mesma estrutura de chaves em todos os idiomas
- Exemplo:
  ```json
  // pt-BR.json
  { "products": { "errors": { "NOT_ENOUGH_STOCK": "Estoque insuficiente" } } }

  // en.json
  { "products": { "errors": { "NOT_ENOUGH_STOCK": "Not enough stock" } } }
  ```

## Regras

- Nunca usar texto hardcoded — sempre i18n
- Ao criar uma nova chave, adicionar em **todos** os idiomas e na API simultaneamente
- Chaves de uso global (botões, validações, status) ficam sob `common.*`
- Chaves específicas de módulo ficam sob `nome_do_modulo.*`
